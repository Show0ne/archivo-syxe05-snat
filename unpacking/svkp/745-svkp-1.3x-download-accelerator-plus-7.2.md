# Introducción

Este tutorial explica el proceso completo de **desempaquetado de SVKP
v1.3x**, un packer que incorpora múltiples técnicas anti‑debug y
anti‑trace.

El análisis se realiza sobre:

Download Accelerator Plus v7.2

Objetivos del tutorial:

-   desempaquetar el ejecutable protegido
-   localizar el OEP real
-   reconstruir la IAT
-   eliminar las técnicas anti‑debug

------------------------------------------------------------------------

# Identificación de la protección

Utilizando detectores de packers como:

-   PEiD
-   RDG Packer Detector

se identifica claramente la protección:

SVKP v1.3x

El ejecutable además incorpora:

-   limitación de trial (30 días)
-   nagscreen

------------------------------------------------------------------------

# Preparación del entorno

Para analizar el ejecutable se utiliza **OllyDbg** con varios plugins.

Plugins utilizados:

-   CmdBar.dll
-   IsDebug.dll
-   OllyDump.dll
-   OllyScript.dll

SVKP utiliza **detección de debugger** mediante la API:

IsDebuggerPresent

Por lo tanto es necesario ocultar el debugger mediante plugin o parche
manual.

------------------------------------------------------------------------

# Problema inicial en OllyDbg

Al ejecutar el programa en OllyDbg aparecen errores como:

-   "Too long SEH chain"
-   "Stack overflow"

Esto ocurre porque el packer utiliza **SEH recursivo** para dificultar
el debugging.

La solución consiste en modificar las excepciones en:

OPTIONS → DEBUGGING OPTIONS → EXCEPTIONS

y manejar manualmente las excepciones durante la ejecución.

------------------------------------------------------------------------

# Desencriptado del código

Durante la ejecución se detecta un loop encargado de desencriptar
código.

El loop realiza operaciones similares a:

-   leer bytes de la sección .code
-   aplicar una clave XOR
-   escribir el resultado en memoria temporal

Esto genera una sección temporal en memoria donde se reconstruye el
código real del programa.

------------------------------------------------------------------------

# Uso de RDTSC como anti‑debug

SVKP utiliza múltiples instrucciones:

RDTSC

para medir tiempos de ejecución y detectar debugging.

Según el análisis del código (página 7 del documento), el packer
contiene gran cantidad de estas instrucciones distribuidas por todo el
código. fileciteturn4file0

Para neutralizarlas se reemplazan con:

xor eax, eax

Esto evita el control temporal utilizado por el packer.

------------------------------------------------------------------------

# Eliminación masiva de RDTSC

Para facilitar el parcheo se genera una lista de todas las direcciones
donde aparece la instrucción.

El procedimiento consiste en:

1.  Buscar todas las instrucciones `rdtsc`
2.  Exportar la tabla de direcciones
3.  Convertir la tabla a formato hexadecimal
4.  Generar un script para parchearlas automáticamente

------------------------------------------------------------------------

# Localización del OEP

Tras eliminar las técnicas anti‑debug se coloca un:

BPM (Memory Breakpoint)

en la sección de código.

Finalmente se localiza el **OEP real** en:

401000

En este punto aparecen también los **stolen bytes** insertados por el
packer, que deberán reconstruirse posteriormente.

------------------------------------------------------------------------

# Reconstrucción de la IAT

Una vez en el OEP se analiza la Import Address Table.

La IAT se encuentra aproximadamente entre:

Inicio: 51A000\
Fin: 51B5A8

Durante el análisis se detectan varias entradas redirigidas hacia
rutinas del packer.

Algunas APIs identificadas:

-   GetVersion
-   GetModuleHandleA
-   GetCurrentProcess
-   GetVersionExA

------------------------------------------------------------------------

# APIs emuladas por el packer

SVKP redirige algunas llamadas a funciones internas que posteriormente
saltan a la API real.

Esto se realiza mediante instrucciones como:

JMP eax

que finalmente terminan en la API verdadera.

------------------------------------------------------------------------

# Dump del ejecutable

Una vez localizado el OEP se realiza el dump con:

OllyDump

Configuración:

-   Get EIP as OEP
-   Dump executable

Posteriormente se utiliza:

Import Reconstructor (ImpREC)

para reconstruir la IAT.

------------------------------------------------------------------------

# Sección adicional para el unpacking

Para completar el proceso se agrega una nueva sección al ejecutable
utilizando herramientas como:

ToPo v1.2

Esta sección almacenará:

-   código auxiliar
-   rutinas de inicialización
-   memoria para emulación de APIs

------------------------------------------------------------------------

# Uso de VirtualAlloc

La rutina de inicialización utiliza la API:

VirtualAlloc

para reservar memoria y cargar el código necesario para emular algunas
funciones del packer.

También se utilizan:

-   GetModuleHandleA
-   GetProcAddress

para localizar dinámicamente las funciones necesarias.

------------------------------------------------------------------------

# Injerto final

Se crea un pequeño **injerto de código** que:

1.  inicializa memoria
2.  corrige la IAT
3.  restaura registros
4.  continúa la ejecución normal

Este injerto se ejecuta antes del OEP original.

------------------------------------------------------------------------

El ejecutable final:

-   queda completamente desempaquetado
-   funciona sin SVKP
-   mantiene su funcionalidad original
